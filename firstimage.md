OK、案3（ブロック連なり編集）から入るのはかなり筋がいい。理由は単純で、TeX文書の多様性（変なマクロ、独自環境、怪しいパッケージ）に対して“壊れにくい”編集モデルを作りやすいから。

ここではあなたの条件を満たす「いちばん強い設計案」を出す：
	•	コードが真実（source of truth）
	•	ブロック編集は“投影”で、対応してないものは生コードブロックで安全に扱う
	•	どんなTeX文書（論文/ノート/テンプレ/ごちゃ混ぜ）でも破綻しない
	•	未来に“ほぼノーコード”へ伸ばせる

⸻

結論：最適設計は「ロスレスAST＋ブロック認識レイヤ＋レンダラ/エディタの分離」

1) コアのデータモデル

(A) ロスレスAST（Concrete Syntax Tree）を持つ

TeXを「意味解析」しようとすると宗教戦争になるので、まずは “文字を失わない構文木” を作る。
	•	入力の空白、改行、コメント、順序、未知コマンドを そのまま保持
	•	解析不能な部分があっても 壊れずに復帰（error recovery）
	•	編集の結果、元のスタイル（インデントや改行）をなるべく保てる

これが「安定性」の背骨。

(B) ブロックはASTの“ビュー”として生成する

ブロックはソースの真実ではなく、ソースから作られる“編集単位”。
	•	例：environment ノード → “Figureブロック”に投影
	•	未知環境 → “Rawブロック”に投影（中身は生コード）

⸻

2) ブロックの種類：最初から「全部対応」を目指さない（が、全て表示できる）

あなたの方針どおりで、最初は“対応ブロック”を少数に絞る。

最初に対応すべきブロック（論文/ノートで強い）
	•	見出し：\section 系
	•	数式：equation/align/gather と $...$（後者はまず“inline math”扱い）
	•	図：figure（画像、caption、label、placement）
	•	表：table/tabular（MVPは“生コード＋簡易表エディタ”のハイブリッドでもOK）
	•	箇条書き：itemize/enumerate
	•	定理系：theorem/lemma/proof（ユーザー定義環境にも拡張可能）

対応してないものは全部これ
	•	Raw Code Block（生コードカード）
	•	折りたたみ可能
	•	その場で編集できる（通常のコードエディタと同じ入力体験）
	•	“対応ブロックに変換”ボタンだけ置く（可能な場合）

これで「どんな文書でも表示できる」＝破綻しない。

⸻

3) ブロック番号・ID設計（めちゃ重要）

「ブロックに番号を付ける」は正しい。さらに 番号＝表示用、ID＝内部用で分けると強い。

表示用番号
	•	画面上だけの B12, B13...
	•	アウトライン/検索で使う

内部用ID（安定の鍵）

ソース編集で位置がズレてもブロックを同一視できる必要がある。

おすすめはこの優先順位：
	1.	既存の \label{...} があればそれをアンカーにする（図・式・節はこれが強い）
	2.	なければ 環境名＋内容ハッシュ（正規化した中身）＋近傍コンテキストでID生成
	3.	それも揺れそうなら、最終手段として tex180用コメントマーカーを埋め込む（任意）
	•	例：% tex180:id=...
	•	ただし「tex180を捨ててもTeXとして成立」が条件

この設計だと、コードファーストでもブロックの同一性が崩れにくい。

⸻

4) 編集の仕組み：常に「パッチ（差分）」でソースに反映

ブロック編集は、ソースを“書き換えるプログラム”になる。ここで事故ると終わる。

原則
	•	ブロック編集 → 最小差分パッチを生成
	•	適用前に差分プレビュー（diff）を出せる
	•	失敗したらロールバック

具体例
	•	Figureブロックで「caption変更」
→ \caption{...} の引数だけを置換するパッチ
	•	数式ブロックで編集
→ 環境内テキストを置換（周囲の \begin...\end は触らない）

“広範囲の整形”をしないのが安定性のコツ。自動整形は別機能（任意）に隔離。

⸻

5) 「対応してないブロックは生コードで表示」＝UI設計

Block Window（案3）はこうすると破綻しない：

左：ブロック列（カード）

各カードは共通で
	•	タイプアイコン（Figure/Equation/Raw…）
	•	タイトル（自動推定：caption / label / 先頭行）
	•	折りたたみ
	•	「ソースを表示」トグル（いつでもコードが見える）

Raw Code Block の扱いが肝
	•	その場で編集できる（コードエディタを埋め込む）
	•	“このブロックを解析してみる”ボタン（環境として認識できたら自動で対応ブロックに昇格）

これで「未知が混ざるTeX文書」でもUXが破綻しない。

⸻

6) 多ファイル・\input・\include の設計

論文は複数ファイルが普通なので、ブロックモデルもそれに適応させる。

おすすめ：
	•	“マスターファイル”を1つ指定
	•	Block Windowは 論理文書ビュー（\input 展開後の順序）で表示
	•	ただし編集は 元ファイルへパッチ適用（パス情報をブロックが持つ）

これで「章ごとにファイル分割」でも自然に編集できる。

⸻

7) 未来の「ほぼノーコード」へ伸ばす仕掛け

最初はコードファーストでOK。でも将来のために、今から“拡張点”を用意する。

ブロック認識・編集をプラグイン化
	•	Recognizer：どのASTパターンをどのブロックにするか
	•	Renderer：ブロックをどう見せるか
	•	Editor：ブロックのUI編集をどうするか
	•	Patcher：編集結果をソースにどう反映するか

これがあると、後から
	•	circuitikz
	•	algorithm2e
	•	minted
	•	custom theorem env
みたいなのを追加しても、コアが壊れない。

さらに、ユーザーが「自分の環境」を登録できると強い：
	•	“この環境は定理として扱え”
	•	“このコマンドは見出しとして扱え”
みたいな設定で、論文テンプレ文化に対応できる。

⸻

いちばんいい設計案まとめ（短く）

ソース真実主義（コードファースト）＋ロスレスAST＋ブロック投影
対応ブロックは少数精鋭、その他は必ず Raw Code Block で安全に表示・編集。
編集は必ず 最小差分パッチ、IDは \label 優先で安定化。
多ファイルは論理ビューで見せて実体にパッチ。

⸻

この設計だと、「いろんなものが入ったTeX文書」を想定しても、壊れ方が綺麗です。
最初に決めるべき次の一手は、MVPで“対応ブロック”を何に絞るか（5〜7種類くらいが現実的）だけ。あとは上の仕組みで自然に増やせます。
